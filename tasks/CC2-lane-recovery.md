# 任务 CC2：工作流"车道"执行与恢复机制

## 目标
借鉴 Claw Code 的 lane（车道）概念，实现工作流的可恢复执行：将工作流拆分为多个可独立执行和恢复的"车道"，支持部分失败后仅重启失败车道，而非整个工作流。

## 文件
- `executor/src/lane.rs`（新建，车道定义和管理）
- `executor/src/lane_executor.rs`（车道执行器）
- `executor/src/checkpoint.rs`（检查点存储，使用 SQLite）
- `executor/src/recovery.rs`（恢复逻辑）
- `compiler/src/lane_splitter.rs`（将工作流图拆分为车道）

## 具体要求

### 1. 车道拆分算法
- 分析工作流 DAG，识别出无数据依赖的独立子图，每个子图作为一个车道。
- 例如：两个 HTTP 请求并行，可放入不同车道；它们的结果合并节点则依赖两个车道完成。
- 输出：车道列表 + 车道间依赖关系。

### 2. 车道执行器
- 每个车道拥有独立的状态机（Pending, Running, Completed, Failed）。
- 使用 `tokio::task` 并发执行可并行的车道。
- 每个车道在执行每个节点前，将当前状态（已执行节点索引、变量快照）写入检查点。

### 3. 检查点与恢复
- 检查点存储在 SQLite 表中：`(workflow_instance_id, lane_id, checkpoint_id, state_blob)`
- 当某个车道失败时，执行引擎仅重启该车道，从最近的检查点恢复（重放后续节点）。
- 其他已完成的车道保持结果不变。

### 4. 恢复接口
- 提供 API `POST /workflow/instance/{id}/recover`，手动触发恢复。
- 学习引擎可自动调用恢复机制处理瞬态失败。

## 测试用例
- 构建一个包含两个并行 HTTP 请求的工作流，其中一个请求超时失败，触发车道恢复后最终成功。
- 验证总执行时间小于串行重试。

## 验收标准
- 车道拆分算法正确识别并行机会。
- 检查点读写开销 < 总执行时间的 5%。
- 恢复后工作流结果与从头执行一致。
